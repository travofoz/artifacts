01 // The Architectural Arena: Alternative Topologies

To understand why this architecture was chosen, we must map out the alternative Multi-Agent design patterns evaluated, stress-tested, and discarded during core layout verification cycles.

Pattern A: Programmatic Wrapper Discarded

A rigid, deterministic script or Python daemon executing string parsing (regex) on workspace files, triggering execution hooks sequentially based on hardcoded line rules.

Failure Vector
Extreme structural brittleness. Attempting to enforce programmatic certainty on non-deterministic AI outputs creates an architectural trap. If a downstream LLM updates its syntax structure, text matches break completely.

Pattern B: Homogeneous Mirroring Discarded

Spawning an instance of a stateful task runner (like iteratr) to oversee a child subdirectory running an identical instance of the same task engine framework loop.

Failure Vector
Split-brain state collisions. Stateful task systems compile graphs into persistent database event meshes on boot. Modifying a running specification checklist mid-flight forces disk states to drift from memory queues, causing index locks.

Chosen Pattern: Hierarchical Supervisor Sovereign Target

A fluid, context-driven Reasoning Engine (Antigravity/agy) holding macro-goals purely in its active memory window, commanding an isolated, stateless child subdirectory via an out-of-band hardware communication gate.

Systemic Rationale
Total isolation of concerns. The supervisor's timeline progress lives as semantic logic in active context rather than a rigid database schema. It can break tasks into dynamic sub-steps, execute diagnostics, and manipulate the child sandbox without risking state machine corruptions.

02 // The Deep "Why" Ledger: Strategic Architectural Pivots

This panel captures the "Edison Logbook"—the precise architectural reasoning behind the structural choices making up this runtime framework, preserving meta-cognition for future iterations.

Pivot 1: From Numeric Compliance Scores to Uncompromising Binary Gates

The Legacy Approach: The critic agent reads written code and outputs a volatile percentage rating (e.g., 85/100), branching steps based on a sliding numerical threshold.

The Rationale For Abandonment
Numeric grading introduces an ambiguous algorithmic gray zone. An LLM scoring code at 94/100 leaves the supervisor guessing which specific 6% failed, and whether the defects are structural or superficial. Shifting to an explicit PASS/FAIL binary gate forces absolute technical accountability. The critic acts as an uncompromising adversary—if a single stub or unhandled route is identified, it drops an automatic failure.

Pivot 2: From Task-List Mutation Diffs to Clean Remediation Cascades

The Legacy Approach: When a build pass fails verification, the supervisor attempts to inject checkboxes or edit task blocks directly inside the running project's tracking documents.

The Rationale For Abandonment
Hacking live specification files behind the back of an active stateful compiler engine shatters the integrity of the data layer. It forces local files to drift from underlying database caches, causing severe syncing mutations. The corrected strategy implements a Multi-Spec Cascade: the supervisor treats the inner tool as a pure black box, outputting completely clean, standalone remediation specs every cycle.

Pivot 3: From Active Status Polling to Kernel-Level OS Locks (inotifywait)

The Legacy Approach: The supervisor triggers the child builder process in the background and continuously makes repetitive API tool turns every few seconds to poll file modifications.

The Rationale For Abandonment
Financial and context exhaustion. Quizzing a multi-billion parameter reasoning model to repeatedly execute loop assertions during a 15-minute software build pass consumes thousands of unnecessary API tokens and litters the context window with junk telemetry. Utilizing inotifywait blocks the running thread at the Linux kernel level, yielding execution completely until tripped for zero tokens.

Pivot 4: The Ephemeral, Context-Blind Critic Gate

The Legacy Approach: The parent supervisor engine acts as its own auditor, evaluating the child subdirectory directly within its long-running session context.

The Rationale For Abandonment
Implicit cognitive bias loop. An agent that has been guiding an execution path across multiple loops develops an internal motivation to bypass minor errors so it can advance its overall goal list. Launching a separate, single-shot, non-interactive shell block for the audit ensures the critic holds zero historical memory, evaluating code with absolute neutrality.

## 2. Core 'iteratr' & Environment Principles
- **Black-Box Principle**: Treat the inner loop engine as an opaque, stateful actor. Never directly edit, modify, or append strings to files that `iteratr` is actively processing.
- **Behavior Controls**: Global execution behaviors are strictly governed by `project/config.yaml` and `project/opencode.json`.
- **Headless Calling Syntax**: Invoke the headless inner loop engine strictly by targeting the current phase specification with the explicit input flag: `iteratr build -s ../specs/current_phase.md --headless`

## 3. Automated State-Machine Loop Protocols
Execute the following steps continuously across all files stored inside `./future_phases/` until the pipeline pool is entirely exhausted:

### Step 3.1: State Initialization & Crash Recovery Check
1. Read `./state.json` to check for an existing execution checkpoint.
   - If an active phase is recorded, resume directly at **Step 3.2** using that file to recover from a previous system crash.
   - If no checkpoint exists, locate the next file in `./future_phases/`, copy it to `./specs/current_phase.md`, and update `./state.json` with the new active target metadata.

### Step 3.2: Asynchronous Background Deployment & Infinite Lock
1. Launch the inner developer agent loop as a background daemon process inside the sub-folder context, piping standard outputs to a local capture stream:
   ```bash
   cd project && iteratr build -s ../specs/current_phase.md --headless > ../child_exec.log 2>&1 & echo $! > ../child.pid
   ```
2. Instantly execute the hardware watcher script to monitor the process:
   ```bash
   ./monitor.sh &
   ```
3. **The Kernel-Level Wait Block (Do Not Time Out)**: Execute this precise command to pass thread control completely to the Linux kernel via filesystem hooks:
   ```bash
   inotifywait -q -e modify monitor.status
   ```
   *CRITICAL RUNTIME DIRECTIVE: This command will lock the shell thread indefinitely (often taking 10 to 20+ minutes) while the inner loop compiles, cross-assembles, and runs tests. You MUST NOT interpret this long blocking period as a terminal hang, error, or session timeout. Stand down your internal execution timer entirely. The command will release naturally the exact millisecond the background gatekeeper updates the status.*

### Step 3.3: The Zero-Bias Isolated Critic Gate
Once the `inotifywait` block releases, read `./monitor.status`. If process death is confirmed, spawn a completely separate, one-shot, non-interactive `agy` call to execute a cold audit. This critic has zero awareness of the supervisor's historical timeline and zero incentive to cheat:
```bash
agy --goal "Act as a brutal, adversarial forensic systems auditor. Analyze the entire codebase inside project/. Verify absolute alignment against the requirements outlined in specs/current_phase.md. You must evaluate using a strict binary PASS or FAIL framework. Write your explicit final verdict in the first line of project/audit_report.md as exactly 'VERDICT: PASS' or 'VERDICT: FAIL', followed by a detailed technical postmortem. CRITIC AUTOMATIC FAILURE MANDATES: You must return a FAIL if you find any placeholder logic, incomplete methods, comments like '// TODO' or '// Stub', hardcoded mock data, thread-safety violations, or unhandled exceptions."
```

### Step 3.4: Control Branching via Remediation Cascades
Analyze the first line within `project/audit_report.md` to process the binary verdict:

- **BRANCH A: VERDICT: PASS**

  2. Wipe the active target fields in `./state.json` to mark the phase cleared.
  3. Cycle clean and advance back to **Step 3.1** to process the next phase.

- **BRANCH B: VERDICT: FAIL**

  2. **The Zero-Mutation Step**: Do not touch, alter, or edit any files inside `./project/` or `./specs/completed/`.   3. Recycle back directly to **Step 3.2** to run a clean remediation patch pass. Limit these cycles to 4 attempts per phase before tripping the circuit breaker.

04 // The Token-Saving Hardware Gatekeeper Code

This companion shell engine runs asynchronously alongside your background processes. It evaluates PID health and detects loop hangs without triggering an expensive LLM API transaction.

FILEPATH: monitor.sh 
#!/bin/bash
# monitor.sh - Zero-Token Hardware Monitor Daemon

CHILD_PID=$(cat child.pid)
LOG_FILE="child_exec.log"
JETSTREAM_LOG="project/.iteratr/logs/jetstream.log"
STAGNATION_TIMEOUT=300 # 5 minutes execution threshold

echo "[*] Telemetry daemon active. Locking onto PID: $CHILD_PID"

while true; do
    # 1. Check if the inner builder process has died naturally
    if ! kill -0 "$CHILD_PID" 2>/dev/null; then
        wait "$CHILD_PID"
        EXIT_CODE=$?
        echo "[!] Execution terminated. Code: $EXIT_CODE"
        
        if [ $EXIT_CODE -eq 0 ]; then
            echo "TRIGGER_LLM_SUCCESS" > monitor.status
        else
            echo "TRIGGER_LLM_CRASH" > monitor.status
        fi
        break
    fi

    # 2. Check if the log files have frozen (Stagnation/Hallucination Loop)
    if [ -f "$JETSTREAM_LOG" ]; then
        LAST_MOD=$(stat -c %Y "$JETSTREAM_LOG" 2>/dev/null)
        CURRENT_TIME=$(date +%s)
        IDLE_TIME=$((CURRENT_TIME - LAST_MOD))

        if [ "$IDLE_TIME" -gt "$STAGNATION_TIMEOUT" ]; then
            echo "[!] CRITICAL: Engine stagnation caught at $IDLE_TIME seconds."
            kill -9 "$CHILD_PID"
            echo "TRIGGER_LLM_TIMEOUT" > monitor.status
            break
        fi
    fi

    # Sleep 15 seconds. Cost: 0 API tokens.
    sleep 15
done
